План: заявки по объявлениям (поиск авто и уведомления владельцам)

Текущее состояние





Объявления (Car): публикация по подписке, поля для сопоставления — city_id, category_id, vehicle_mark_id, vehicle_model_id (entities.py).



Справочники: категории, марки/модели, города — API GET /dictionaries, GET /dictionaries/model/{marka_id} (dictionaries.py).



Город пользователя: только в сторе на фронте (store.ts); в БД у User есть только address.



Подписка владельца: проверка через get_active_subscription_for_owner(db, owner_id) (subscriptions_service.py); owner_id = client_cars.user_id = автор объявления car.author_id.



1. Модели и БД (backend)

Новые таблицы в back/app/models/entities.py:





Application (таблица applications) — заявка от клиента:





id, user_id (FK users), city_id, category_id, **vehicle_mark_id (FK dictionaries, nullable), **vehicle_model_id (FK dictionaries, nullable)



Для легковых автомобилей: марка обязательна (или опциональна — уточнить по справочнику категорий), модель опциональна (можно не выбирать, но возможность выбора оставить).



Для остальных категорий (например, спецтехника): марка и модель опциональны (можно не выбирать ни марку, ни модель).



requested_at (DateTime — «на когда»)



message (Text, макс. 1000 символов)



status: ACTIVE | COMPLETED | REJECTED



views_count (Integer, default 0) — общее количество просмотров заявки



create_date, update_date, completed_at (DateTime, nullable)



Связи: User, Dictionary (city, category, mark, model), изображения — через существующую таблицу Image (entity_type = 'APPLICATION', entity_id = application.id); в entities.py добавить relationship на Application к Image по этим полям (как у Car).



ApplicationCar (таблица application_cars) — связь заявка ↔ объявление:





id, application_id (FK applications), car_id (FK cars)



Уникальность пары (application_id, car_id)



Поле owner_read_at (DateTime, nullable) — когда владелец объявления прочитал заявку (для счётчика непрочитанных)



Связи: Application, Car



ApplicationSelectedCar (таблица application_selected_cars) — выбор объявлений при завершении заявки:





id, application_id (FK applications), car_id (FK cars)



Уникальность пары (application_id, car_id). Заполняется только при переходе в COMPLETED: автор заявки выбирает одно или несколько объявлений из списка совпадений, выбор сохраняется здесь.



Связи: Application, Car

Настройки уведомлений и город — расширить модель User (entities.py):





notify_by_email (Boolean, default True) — отправлять уведомления на почту (поле показывать в настройках только если у пользователя заполнен email).



notify_by_whatsapp (Boolean, default True) — отправлять уведомления в WhatsApp; опционально: функционал «уведомления в WhatsApp» доступен только при активной подписке.



**city_id** (FK dictionaries, NOT NULL) — выбранный город пользователя. Дефолт при регистрации — id города «Алматы» из словаря. При смене города в шапке сайта сохраняется сюда и используется для фильтрации списков объявлений и заявок (см. п. 3 и раздел «Город» ниже).

Логика сопоставления при создании заявки: выборка Car с status == "PUBLISHED", delete_date.is_(None), city_id = application.city_id, category_id = application.category_id, и по марке/модели: (app.vehicle_mark_id is null or car.vehicle_mark_id = app.vehicle_mark_id) и (app.vehicle_model_id is null or car.vehicle_model_id = app.vehicle_model_id). Таким образом, если в заявке не указана модель — подходят все объявления категории (и при необходимости марки); если не указаны марка и модель — по городу и категории. Для каждой подходящей машины создаётся запись ApplicationCar. После создания заявки — отправка уведомлений владельцам (email/WhatsApp по настройкам пользователя).

Фото к заявке  





Использовать существующую таблицу Image (entities.py): entity_type = 'APPLICATION', entity_id = application.id. В модели Application добавить relationship к Image (аналогично Car). Схему Image не менять — только новый тип сущности. Загрузка через текущий поток (например, Cloudinary + POST /images или multipart в POST /applications). Автор заявки может прикрепить несколько фото (например, что нужно поднять для спецтехники); владелец и автор могут просматривать фото в карточке заявки и в списках (миниатюра в карточке).

Обновить back/app/init_db.py: импорт новых моделей и создание таблиц.



2. API заявок (backend)

Новый роут back/app/api/v1/routes/applications.py, подключить в router.py с prefix="/applications".





POST / — создание заявки (любой авторизованный пользователь):





Тело (multipart или JSON + отдельная загрузка): city_id, category_id, **vehicle_mark_id (опционально), **vehicle_model_id (опционально) — для категории «легковые» модель можно не передавать; для других категорий можно не передавать марку и модель; requested_at (ISO datetime), message (до 1000 символов), **images** (опционально — один или несколько файлов, либо уже загруженные image_id/url через текущий API изображений).



Валидация, создание Application со статусом ACTIVE, привязка изображений к заявке (Image с entity_type APPLICATION, entity_id = application.id), поиск подходящих Car по правилам с учётом nullable mark/model, создание ApplicationCar. Отправка уведомлений владельцам.



Ответ: заявка + список совпавших объявлений + массив images (url и т.п.) для просмотра.



GET / — «мои заявки»:





В ответе для каждой заявки: данные заявки (в т.ч. views_count), первое фото или массив фото для превью, количество совпадений.



GET /{application_id} — одна заявка с списком объявлений:





Проверка доступа: автор заявки или владелец одного из объявлений по этой заявке. Если запрос от владельца объявления — увеличить Application.views_count и при необходимости выставить owner_read_at в ApplicationCar.



Возврат заявки (в т.ч. views_count, все прикреплённые изображения), массив объявлений. Для COMPLETED — выбранные объявления.



PATCH /{application_id} — смена статуса:





REJECTED: может выставить автор заявки («Отменить заявку») или владелец любого из объявлений по этой заявке («Отклонить»). Тело: status: "REJECTED".



COMPLETED: может выставить только автор заявки. Тело: status: "COMPLETED", **selected_car_ids** (массив id объявлений из совпадений по этой заявке; обязательно хотя бы один, все должны входить в ApplicationCar для данной заявки). Бэкенд сохраняет выбранные объявления в application_selected_cars, выставляет completed_at, меняет статус на COMPLETED. Двустороннее подтверждение не делаем — только автор подтверждает, выбирая объявление(я).



GET /to-my-ads — заявки, относящиеся к моим объявлениям:





Фильтрация по городу: application.city_id = current_user.city_id. Объявления текущего пользователя: Car.author_id == current_user.id; выборка заявок через ApplicationCar.car_id in (my_car_ids).



Для каждой заявки: данные заявки (в т.ч. views_count), изображения заявки (для просмотра, например что нужно поднять), контакт заявителя — только при активной подписке; иначе «Купить подписку, чтобы увидеть контакты». У владельца только кнопка «Отклонить».



GET /to-my-ads/count — количество непрочитанных заявок к моим объявлениям (по current_user.city_id и моим машинам), для бейджа в шапке.



POST /to-my-ads/mark-read — отметить заявки как прочитанные при открытии вкладки «Заявки к моим объявлениям».



GET /other — другие заявки (не по моим объявлениям): фильтр по application.city_id = current_user.city_id. Поля: город, категория, марка/модель, дата, сообщение (без контактов). Опционально фильтр по категории.

Схемы запросов/ответов — back/app/schemas/applications.py.



3. Точка входа «Найти авто» и фильтрация по городу

Город пользователя (User.city_id)  





В БД у пользователя хранится city_id (обязательно, дефолт — Алматы из словаря). При смене города в шапке сайта: сразу вызывать PUT /auth/me (или PATCH) с новым city_id и обновлять локальный стор по имени города из словаря — выбранный город везде совпадает.  



Все запросы к БД по объявлениям и заявкам учитывают выбранный город:





GET /cars (каталог, главная) — фильтр Car.city_id = current_user.city_id (из профиля/токена).



GET /applications/to-my-ads, GET /applications/to-my-ads/count, GET /applications/other — фильтр по Application.city_id = current_user.city_id.



В форме заявки город по умолчанию — из профиля пользователя (user.city_id); при смене города в шапке значение синхронизируется с профилем. Для этого GET /auth/me и PUT /auth/me должны возвращать и принимать city_id (id из словаря).

Точка входа «Найти авто»





Главная front/app/(public)/page.tsx: рядом с «Выбрать авто» (каталог) добавить кнопку «Найти авто» (или блок поиска), ведущую на страницу подачи заявки (например, /find или /requests/new). Для неавторизованных — редирект на логин с returnUrl на эту страницу.



Опционально: в front/app/(public)/catalog/page.tsx добавить кнопку «Не нашли? Оставить заявку» → та же страница подачи заявки.



4. Страница подачи заявки (форма)

Новая страница (например, front/app/(public)/find/page.tsx или front/app/(public)/requests/new/page.tsx):





Поля:





Город — выбор из словаря CITY; дефолт — из профиля пользователя (user.city_id).



Категория — из словаря CATEGORY (Авто, Спецтехника и т.д.). От выбранной категории зависит обязательность марки/модели.



Марка — из словаря MARKA; для категории легковые автомобили — показывать и дать возможность выбора; для остальных категорий (например, спецтехника) — опционально (можно не выбирать).



Модель — после выбора марки подгружать GET /dictionaries/model/{marka_id}; для легковых — опционально (можно не выбирать); для остальных категорий — опционально (если марка не выбрана, поле не показывать или показывать неактивным).



Дата и время — один контрол даты+времени; формат для API — ISO.



Сообщение — текст до 1000 символов, с счётчиком.



Фото — возможность прикрепить одно или несколько изображений (например, что нужно поднять для спецтехники). Загрузка при отправке формы (multipart) или через существующий API изображений; отображать превью и возможность просмотра в карточке заявки.



Кнопка «Отправить заявку»:





POST на новый API создания заявки.



После успеха — переход на страницу «Мои заявки» с уведомлением и показом результата «Найдено N объявлений» со ссылкой на список объявлений по этой заявке (или сразу показать список на той же/следующей странице).

Форма доступна только авторизованным; иначе — редирект на логин.

6. Страница «Заявки» в профиле (вкладки: Мои заявки / К моим объявлениям / Другие заявки)

Новая страница (например, front/app/profile/requests/page.tsx) с вкладками:





«Мои заявки»





Форма создания заявки (город по умолчанию из user.city_id, категория, марка, модель, дата/время, сообщение до 1000 символов). POST /applications → обновление списка и показ «Найдено N объявлений» со ссылкой на список.



Список заявок: активные (ACTIVE) и история (COMPLETED, REJECTED). В карточке заявки показывать количество просмотров (views_count) — «Просмотров: N». Для каждой карточки: категория, марка/модель, дата/время, сообщение (сокращённо), статус, кнопка «Список объявлений». Для ACTIVE — кнопка «Завершить»: по нажатию открывается список объявлений по этой заявке, пользователь обязательно выбирает одно или несколько объявлений (чекбоксы), затем отправка PATCH с status: "COMPLETED" и selected_car_ids; кнопка «Отменить заявку» (PATCH REJECTED).



«Заявки к моим объявлениям»





Список заявок по объявлениям текущего пользователя (GET /applications/to-my-ads). В карточке заявки — количество просмотров (views_count) при необходимости. Для каждой заявки: категория, марка/модель, дата, сообщение; контакты заявителя — только при активной подписке, иначе «Купить подписку, чтобы увидеть контакты». Для ACTIVE у владельца только кнопка «Отклонить» (PATCH REJECTED); кнопки «Сделка состоялась» у владельца нет — завершает только автор заявки. При открытии вкладки — вызов API прочитанности (сброс счётчика в шапке). Внизу — вся история (фильтр по статусу).



«Другие заявки»





Список заявок, не относящихся к объявлениям пользователя (GET /applications/other). Только просмотр: город, категория, марка/модель, дата, сообщение — без контактов.

Завершение заявки (только автор)  





«Завершить» доступна только автору заявки (в «Мои заявки»). По нажатию показывается список объявлений по этой заявке; пользователь обязательно выбирает одно или несколько объявлений (с кем сделка состоялась), затем отправляется PATCH с status: "COMPLETED" и selected_car_ids. Выбор сохраняется в application_selected_cars. Двустороннее подтверждение не делаем — только автор подтверждает. У владельца объявления кнопки «Сделка состоялась» нет, только «Отклонить».

Обновление списка заявок (real-time)  





Чтобы клиент видел, что заявку уже завершил кто-то другой (или отклонил владелец), без ручного обновления страницы, возможны варианты: (1) периодический опрос (polling) списка заявок каждые N секунд на странице «Мои заявки» / «Заявки к моим объявлениям»; (2) WebSocket/SSE для обновления статусов. Если real-time не делаем в первой версии: при открытии страницы всегда загружать свежие данные; при нажатии «Завершить» или «Отклонить» на бэкенде проверять, что статус ещё ACTIVE, и при уже COMPLETED/REJECTED возвращать ошибку с сообщением «Заявка уже завершена» / «Заявка отклонена». Рекомендация: минимум — проверка на бэкенде; опционально — polling раз в 30–60 сек на странице заявок.



7. Уведомления (email и WhatsApp)





Событие: при создании заявки (POST /applications) для каждого совпавшего объявления владельцу отправляется уведомление (новая заявка по его объявлению).



Каналы: email и/или WhatsApp (ВЦ). Отправка по настройкам пользователя-владельца:





если включено notify_by_email и у пользователя есть email — письмо;



если включено notify_by_whatsapp и у пользователя есть phone_number — уведомление в WhatsApp (интеграция через API WhatsApp Business или ссылка wa.me; детали выбора провайдера — отдельно).



Настройки в профиле (например, в существующей вкладке «Настройки» profile/page.tsx или в разделе «Уведомления»):





«Отключить уведомления на почту» — чекбокс/переключатель; показывать только если у пользователя заполнен email. Сохранять в User.notify_by_email.



«Отключить уведомления в WhatsApp» — чекбокс/переключатель. Опционально: показывать только пользователям с активной подпиской (иначе подсказка «Доступно по подписке»). Сохранять в User.notify_by_whatsapp.



Бэкенд при отправке уведомления проверяет эти флаги и наличие email/телефона; при необходимости — проверка подписки для разрешения отправки в WhatsApp.

9. Уточнения и замечания





Именование: таблица applications, связки application_cars и application_selected_cars; статусы ACTIVE, COMPLETED, REJECTED.



Город: в таблице users добавлен обязательный city_id (дефолт — Алматы из словаря). При смене города в шапке — сохранение в профиль (PUT /auth/me) и обновление стора; все списки объявлений и заявок фильтруются по current_user.city_id.



Просмотры заявки: в карточке заявки отображается общее количество просмотров (views_count); при открытии заявки владельцем счётчик увеличивается.



Завершение: только автор заявки может нажать «Завершить»; при этом обязательно выбирает одно или несколько объявлений из списка совпадений (selected_car_ids → сохраняются в application_selected_cars). Владелец может только «Отклонить» (REJECTED). Двустороннее подтверждение не делаем.



Real-time: в первой версии достаточно проверки на бэкенде при действии (если заявка уже не ACTIVE — вернуть ошибку); опционально — polling списка заявок каждые 30–60 сек на странице заявок, чтобы без перезагрузки видеть смену статуса.



Уведомления: реализация отправки в WhatsApp выносится в отдельную задачу; в плане заложены поля User и настройки в UI.



Другие заявки: только просмотр (без контактов), для понимания спроса.



Фото к заявке: через таблицу Image (entity_type APPLICATION); автор может прикрепить несколько фото при создании заявки; просмотр в карточке заявки и в списках (превью).



Марка и модель опциональны: для категории «легковые автомобили» — модель можно не выбирать; для остальных категорий (например, спецтехника) — марку и модель можно не выбирать; сопоставление с объявлениями по правилу «null = любое значение» по соответствующим полям.
